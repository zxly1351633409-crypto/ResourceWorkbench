## 2026-07-11 v0.3.0 安全语义补丁：不自动解压与正式移动二次确认

### 为什么改

- 卡片右键仍写着“解压后深度分析”，但工作线程用 `if False` 跳过解压；确认文案又先描述会解压、随后被“不解压”覆盖，界面与实际行为互相矛盾。
- 移动预演仍会计算来源、实际目标、文件数和容量，但 GUI 的确认函数只检查 `ok` 就直接返回，导致正式移动没有展示计划，也不再要求输入 `MOVE`。

### 改了什么

- 普通 `AnalyzeWorker` 固定忽略旧的自动解压参数，扫描目标始终是用户输入的原路径；卡片重新分析删除 `if False` 解压死分支，压缩包只按一个外层文件扫描，`inspect_archives=False`。
- 压缩包右键入口改为“分析压缩包外层（当前不解压）”；文件夹入口也明确遇到压缩包不会解压。确认、进度、详情和完成状态统一使用“重新分析（未解压）”语义。
- 移动确认先逐项展示来源、实际目标、文件数/容量及总计。正式计划先确认预览，再要求完整输入大写 `MOVE`；取消、关闭或输错均返回且不调用执行函数。测试计划保留普通确认。
- 单卡、批量和审阅队列共用同一确认函数；移动成功后的数量/容量复核与可撤销日志保持不变。

### 怎么验证

- 新增回归覆盖：旧参数无法恢复自动解压、压缩包重新分析仍扫描原压缩包且不检查内部、右键文案无“解压后”歧义。
- 新增回归覆盖：正式计划展示来源/目标/摘要并要求 `MOVE`；预览取消或确认文字错误返回 false；测试计划不要求 `MOVE`；真实单卡取消后来源和卡片均保持原状。
- 验证命令：`python -m compileall -q src tests` 与 `python -m unittest discover -s tests -v`。

## 2026-07-11 v0.3.1：移动安全、异步资源树、SpeedTree 与后台运行数据维护

### 为什么改

- 历史需求复核发现，慢 NAS 的树节点枚举和每个子目录探测仍在 GUI 线程，第三层以后虽然能显示，但展开时仍可能卡住；400 项上限还会静默截断。
- 旧安全补丁修复了移动确认和压缩包文案，但尚未进入新的独立版本/分发哈希。
- SpeedTree 只有扩展名识别，Bark/Clusters 的 AO、Normal、Gloss、Opacity 贴图可能压过工程真实预览。
- 原维护只覆盖预览、报告和移动日志；staging、资源索引、复核/重命名/上传历史及 SQLite 空闲页仍可增长，而且启动自动清理在 GUI 线程执行。

### 改了什么

- 资源树根加载、展开和刷新改为后台 QRunnable；结果使用 generation、request id 和 offset 三重校验。单页 400 项，提供“继续加载”；第二页失败保留已加载内容并按原 offset 重试。
- 为图片预览、目录签名和资源树任务增加活动注册表；窗口关闭使 generation 失效，`main()` 在 QApplication 仍存活时 drain 四个线程池，修复退出阶段 `0xC0000409`。
- `speedtree_preview.py` 只读识别 `.spm` 和二进制 `.srt`，精确匹配同目录 `preview.png` / `<stem>_preview.png`，排除材质通道图。没有真实渲染时只在缓存生成明确非渲染占位；历史 GUI 抢焦点脚本不被复制或自动调用。
- staging manifest 增加 complete/completed_at 和活动标记。只有完整、源包仍保留、无活动标记且过期的批次可清理。
- 移动日志只清理已撤销历史；资源索引可重建；复核/重命名/上传只清理明确完成终态；人工标签/备注只 VACUUM 不删行。
- `RuntimeMaintenanceController` 在 QThread 中执行自动维护、手动 dry-run 和 apply；重复启动被拒。维护中关闭窗口会等待当前操作自然结束，不强杀删除/SQLite。
