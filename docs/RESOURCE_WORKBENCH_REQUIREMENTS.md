# 资源入库工作台：需求规范

> 版本 0.3.1（2026-07-11）说明：本节是当前实现的最高优先级规范；下方早期设计仍保留作历史和长期路线参考。若内容冲突，以本节为准。

## 0. 当前产品定义与验收状态

ResourceWorkbench 是“资源入库前的审阅工作台”，不是逐文件归档器。它把用户指定的本地资源批次或网页链接转为可视卡片，辅助翻译、判断、选择目标和可回滚移动。资源边界默认以外层资源文件夹为准。

### 0.1 当前强制交互

- 顶部只有统一输入框、分析按钮和分析时出现的取消按钮。
- 输入框支持文件夹、压缩包、分号/换行分隔的多个来源、`http/https` 链接。
- 只有点击分析或按 Enter 才运行；`editingFinished` 禁止触发深度分析。
- 左侧资源库点击只更新资源库视图，待整理结果必须保留。
- 长提示放在右下角小状态条；分析状态放在卡片区域。

### 0.2 当前压缩包策略

- `ScanConfig.inspect_archives` 默认 `False`。
- 普通分析不列压缩包内部目录，也不依据 `archive_subresources` 拆卡。
- 普通分析与卡片重新分析均不得自动解压；界面文字必须明确“当前不解压”，不能承诺代码实际不会执行的流程。
- 贴图/通道/格式目录（textures、maps、normal、roughness、basecolor、fbx、blend 等）永不作为独立资源边界。
- 未来“深度读取压缩包”只能是明确高级选项，不能改变默认行为。
- 解压、移动和删除运输包仍必须有校验与可追溯记录。

### 0.3 当前 UI 与主题

- 视觉方向只借鉴 cc-haha 的轻量工具栏、低噪声边框、统一按钮尺寸和明确 hover/selected；不复制其源码、logo 或图片。
- 左侧资源库和中间卡片墙都使用独立 box 容器。
- 主题采用 Blender 式语义角色：窗口、画布、面板、侧栏、卡片、按钮、输入框、正文、弱文字、边框、图标、悬停和选中态可独立覆盖。
- 工作台头部不显示版本号；版本仅在左下角以弱文字显示。
- 设置和颜色选择器的标准按钮必须为中文；自定义配色提供语义分组、预设、单项恢复和实时预览。

### 0.4 深层资源库浏览与同步

- 点击任意层级目录时，当前目录必须独立建立浅索引，不受启动预载深度限制。
- 有子目录时以子目录为卡片；真正叶目录以直系媒体文件为卡片，忽略 `Thumbs.db`、`desktop.ini`、`.DS_Store`。
- 实际媒体类型优先于大类路径；纯视频目录不能因位于“照片”树下而显示照片。
- 图片解码和视频抽帧不得在 GUI 主线程执行；视频并发上限应低于普通图片。
- 左侧支持在资源库根内安全新建文件夹；拒绝越界、保留名、非法字符和重复目录。
- 监听根、当前、已展开目录；NAS 监听不可靠时使用后台非递归签名轮询兜底，并保留树的展开/选中状态。
- 目录枚举与子目录探测必须在后台运行；旧 generation/request/offset 结果不得覆盖新根或刷新。单层超过 400 项必须分页并提供继续加载，失败保留已加载内容并允许按原 offset 重试。

### 0.6 数据增长与学习

- 每次校验通过的移动记录卡片特征与真实目标目录；推荐时以有限权重加入历史命中。
- 已撤销和校验失败的记录不参与学习。
- 移动日志只清理明确已撤销的终态历史；仍可撤销、撤销失败和未知状态永久保护。
- 预览缓存按最大容量/最长天数清理；分析报告也限制数量和时间。
- staging 只有同时满足“完整 manifest、源包仍保留、无活动标记、超过保留期”才可删除。
- 资源索引可按时间/条数重建；复核、重命名、上传日志只清理明确完成的终态历史。人工标签/备注只允许 VACUUM，不删除数据行。
- 检查、预演和执行必须放在后台线程；维护中关闭窗口要等待当前数据库/删除操作自然结束，不能强杀。
- 清理只作用于工作台自有数据，绝不删除源资源。

## 1. 项目目标

将用户当前的资源整理流程整合为一个桌面工作台：

```text
指定批次
-> 解压/递归解压
-> 识别资源边界
-> 生成卡片
-> 翻译/命名建议
-> 推荐 Z 盘分类
-> 用户确认/纠错
-> 整理封面与工程目录
-> 移动到 Z:\整合——资源管理
-> 上传到 115 同路径
-> 清理运输压缩包
```

核心原则：

- 不自动扫全盘。
- 不越过用户决策。
- 不确定就标记需确认。
- 移动和删除必须可撤销或有校验。
- 用户的分类习惯优先于通用分类法。

## 2. 用户现有分类根目录

本地资源库：

```text
Z:\整合——资源管理
```

115 网盘中有同名目录结构，上传时应使用相对路径映射。

主要顶层分类：

```text
A Alphas
B Blender笔记
B 笔刷
C 材质
J 教程
M 模型
U UE
Z zb brush
Z 照片
```

## 3. 管理单位

管理单位不是单张图片，也不是单个文件，而是“资源文件夹”。

层级：

```text
整理批次
  ├─ 资源卡片 A
  ├─ 资源卡片 B
  └─ 资源卡片 C
```

一个外层压缩包可能包含多个资源卡片。

## 4. 压缩包策略

### 4.1 当前预览阶段

普通分析不读取压缩包目录。只有用户明确开启未来的“深度读取压缩包”高级选项时，才可以不完整解压、只读取目录。

用途：

- 快速判断类型
- 判断是否为大合集
- 生成初步卡片
- 抽取单张预览图

### 4.2 正式入库阶段

必须解压到临时工作区后再入库。

不能直接把运输压缩包移动到 Z 盘当作资源。

正式移动对象应该是：

- 解压后的资源文件夹
- 或明确被用户标记为“资源内容压缩包”的文件/文件夹

### 4.3 压缩包类型

需要区分：

1. 运输压缩包  
   只用于下载和传输，入库成功后可清理。

2. 资源内容压缩包  
   产品本身就是 zip/rar/7z 或内部安装包，需要保留。

3. 分卷压缩包  
   例如 `.part1.rar`、`.part2.rar`，只能从入口卷解压。

4. 未知压缩包  
   交给用户确认。

## 5. 密码规则

用户说明：密码一般在压缩包所在文件夹名 `_` 后面的字符数字。

密码候选顺序：

1. 空密码
2. 父文件夹 `_` 后面的字符
3. 父文件夹里的明显数字串
4. 压缩包名里的明显数字串
5. 用户手动输入的批次密码

示例：

```text
xxx_123456\file.rar -> 123456
科幻29-20230512.scifi\科幻29.part1.rar -> 20230512
```

## 6. 资源类型识别

软件根据这些信息判断：

- 文件扩展名
- 压缩包目录
- 文件夹名
- README/说明文档
- 图片/视频/PDF 数量
- 现有 Z 盘分类树
- 用户历史纠错记录

不依赖 DeepSeek 做图片识别。

DeepSeek 只用于：

- 文本翻译
- 名称摘要
- 标签建议
- 从候选分类中选择

## 7. 分类推荐

分类推荐必须参考 `Z:\整合——资源管理` 现有结构。

示例规则：

```text
sci-fi / futuristic / robot / mech -> M 模型\K 科幻
robot / mech -> M 模型\K 科幻\J 机甲
ammo / bullet / rifle / pistol -> M 模型\K 科幻\Q 枪支 或现代/军事枪支分类
city / factory / environment -> M 模型\K 科幻\C 场景
rocket / spaceship / vehicle -> M 模型\K 科幻\Z 载具
basalt / rock / stone -> Z 照片\Z 自然\S 石头
lake / water -> Z 照片\Z 自然\S 水
```

推荐结果应显示原因，不允许只给路径。

## 8. 大合集拆分

普通分析的默认策略：只按磁盘外层真实资源目录建卡，不依据压缩包内部结构拆分。下述“大合集拆分”只适用于未来明确开启的深度读取模式。

判断依据：

- 压缩包内存在多个同级产品目录
- 每个目录有独立 `file.jpg`、`Assets` 或资源包
- 子目录数量大于等于 3

示例：

```text
科幻29-20230512.scifi
  ├─ 300 Blackout Ammo...
  ├─ 40 Futuristic Sci-fi City...
  ├─ 5 Sci-Fi trim sheets...
  └─ ...
```

应该生成多张卡，而不是一张“科幻29”卡。

## 9. 预览图

预览来源优先级：

1. 资源文件夹外层封面图
2. 子资源目录中的 `file.jpg`
3. 压缩包内第一张代表图
4. 失败时显示无预览，不阻塞流程

预览图只缓存到：

```text
workbench_data\previews
```

不修改源资源。

## 10. 翻译功能

尚未实现，但必须做。

设计：

- API：DeepSeek
- 输入：资源原名、压缩包目录样例、说明文本
- 输出：
  - 中文名
  - 英文原名
  - 中英并存建议名
  - 标签
  - 简短说明

命名风格参考用户现有习惯：

```text

```

翻译必须可人工确认。

## 11. 入库移动

正式入库时必须先经过用户确认。

移动前需要显示：

- 资源卡片名
- 预览图
- 目标分类
- 移动对象
- 是否保留内部压缩包
- 是否整理为“封面图 + 工程”

移动后需要记录日志：

- 源路径
- 目标路径
- 文件数量
- 总大小
- 时间
- 操作状态

## 12. 115 上传

上传不是第一阶段。

设计建议：

- 优先 115 官方开放平台 API
- 本地路径转换为相对路径
- 云端使用同样相对路径
- 上传成功后标记备份状态
- 上传失败进入重试队列

## 13. 安全与回滚

所有高风险操作必须有确认：

- 移动
- 删除
- 上传
- 批量改名

默认删除进入回收站，不永久删除。

建议保留操作日志和撤销能力。
